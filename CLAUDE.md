# Pâté → FoodSave CRM: автоматизация загрузки товаров

## Что это

Инфраструктура-прослойка между WhatsApp (ежедневные сообщения с акциями по точкам Pâté)
и CRM FoodSave, чтобы не вбивать товары руками через форму (~1 час в день).

Сейчас процесс: бариста/менеджер каждой точки пишет в WhatsApp текст вида

```
Пате Туран
Тарт абрикос 5160 вместо 8600
Профитроли 3840 вместо 6499 2шт
```

Нужно превратить это в карточки товара в CRM (POST /api/products из `box-endpoint.md`):
название, описание, цена/скидка, фото, дата (срок годности = сегодня+1 по умолчанию),
категория = "Кондитерские изделия" (фиксированная), статус = AVAILABLE.

Это пока чисто локальный инструмент под 5 точек Pâté (не универсальный пайплайн).

## Файлы

| Файл | Назначение |
|---|---|
| `catalog_paste.json` | Мастер-каталог: 77 товаров, для каждого `canonical` имя + `aliases` (варианты как пишут баристы) + `description` + `sizes` (цена/фото/`detail` с размерами на каждый вариант мини/большой/порция/...) |
| `images/` | Фото товаров, имена файлов транслитом (важно — см. "Грабли" ниже) |
| `parse_daily.py` | Парсит текст из WhatsApp → матчит с `catalog_paste.json` (fuzzy по токенам, нечувствительно к порядку слов) → считает скидку (если "вместо" нет — берёт дефолт 40%, `DEFAULT_DISCOUNT_PCT`) → отдаёт JSON карточек. Помечает `needsReview` если матч слабый или скидка предположена |
| `upload_to_crm.py` | Берёт JSON от `parse_daily.py` → логинится в CRM → резолвит `categoryId`/`storeId` по имени через `/api/categories/active` и `/api/stores/active` (без хардкода id) → заливает фото `/api/upload/image` → создаёт товар `/api/products`. Есть `--dry-run` и `--force` (включить needsReview) |
| `config.example.json` | Шаблон конфига для `upload_to_crm.py` (логин/пароль/алиасы точек) — копировать в `config.json`, в git не класть |
| `extract_images.py` | Разовый скрипт: вырезал фото из исходного PDF-каталога (`Каталог_Pate-2.pdf`) под уже заданный список товаров по страницам (`PAGE_ITEMS`) |
| `extract_catalog.py` | **В разработке.** Полностью автоматический парсер PDF → текст (название/описание/размеры/цены) + фото → `catalog_auto.json`. Цель — не транскрибировать каталог руками при каждом обновлении PDF. Работает на страницах 1-6 (сытные пироги, киш, тарты), дальше не протестировано |

## Как прогнать пайплайн руками (когда всё доделано)

```bash
pip install requests pymupdf pytesseract pillow --break-system-packages
# + системный tesseract-ocr с языком rus (см. "Грабли")

# 1) текст из WhatsApp → raw_messages.txt
python3 parse_daily.py raw_messages.txt catalog_paste.json > result.json
# смотрим stderr — там список needsReview позиций, проверяем глазами

# 2) конфиг (один раз)
cp config.example.json config.json   # вписать email/password/store_aliases

# 3) проверка плана
python3 upload_to_crm.py result.json --config config.json --dry-run

# 4) реальная отправка
python3 upload_to_crm.py result.json --config config.json
# report.json — что создалось/упало
```

## Грабли, на которые уже наступили (важно не наступить повторно)

1. **Кодировка имён файлов в zip на Windows.** UTF-8 имена (кириллица) в zip
   разваливаются в проводнике Windows (CP437/CP1252 mojibake). Решение: ВСЕ имена
   файлов картинок — транслит латиницей (см. функцию `sanitize()` / `translit()`
   в скриптах). Если будешь генерировать новые файлы — не забывай транслит,
   иначе кириллица в `images/` снова всё сломает на винде.

2. **Битый cmap у жирного шрифта заголовков в PDF.** В `Каталог_Pate-2.pdf`
   обычный текстовый экстракт (`page.get_text()`) для **жирных заголовков товаров**
   выдаёт искажённые буквы: "куриШей" вместо "куриЦей", "Кищ" вместо "Киш",
   "говЉдиной" вместо "говЯдиной". Это не баг экстракции — у встроенного
   bold-шрифта в самом PDF нестандартный cmap (на глаз в Acrobat/просмотрщике
   всё ок, потому что рендер идёт по глифам, а не по cmap-маппингу в Unicode).
   Обычный (не жирный) текст — описания, цены, детали размеров — экстрактится
   нормально, проблема только в заголовках.
   **Решение:** название товара читаем через OCR (tesseract, `lang=rus`,
   `--psm 6`) по вырезанному из PDF изображению bbox-а заголовка, а не через
   текстовый слой. Реализовано в `extract_catalog.py` (`ocr_header()`).
   Нужен `rus.traineddata`:
   ```bash
   mkdir -p /tmp/tessdata
   curl -fsSL -o /tmp/tessdata/rus.traineddata \
     https://raw.githubusercontent.com/tesseract-ocr/tessdata_fast/main/rus.traineddata
   export TESSDATA_PREFIX=/tmp/tessdata
   ```

3. **Скрытые дублирующиеся слои картинок.** На некоторых страницах PDF одна и
   та же картинка встречается дважды с почти одинаковым `bbox` (видимо, дизайнер
   продублировал слой). Наивный матчинг "заголовок → ближайшая по Y картинка"
   из-за этого ошибается (брал не ту картинку). Решения:
   - в `extract_images.py` — пришлось руками визуально проверять (`view` на
     сохранённые jpg) и фиксить три страницы (6, 15, 17);
   - в `extract_catalog.py` добавлена дедупликация картинок с разницей Y < 5pt
     (берём ту, что больше по `size`) — но это эвристика, не 100% гарантия,
     **визуальная проверка результата всё равно нужна** после любого
     массового прогона на новом PDF.

4. **Незакрытый баг: аномальная позиция цены у "одноценовых" товаров.**
   У товаров без вариантов размера (например "Мясной пирог", одна цена 5700 ₸)
   bbox текста цены в PDF лежит у верхней границы страницы (`y≈73`), хотя
   визуально цена находится в самом низу карточки (`y≈700+` на полноразмерной
   странице). Для товаров с 2-3 вариантами (мини/большой) такого сдвига нет —
   там цены корректно лежат в полосе под описанием. Похоже на проблему
   трансформации матрицы для отдельного Form XObject именно у "одиночных"
   прайс-плашек. **Не доделано** — `extract_catalog.py` сейчас не находит
   цену для таких товаров (см. warning при прогоне). Возможные пути:
   - искать "осиротевшие" цены (не попавшие ни в одну полосу между
     заголовками) и приписывать их ближайшему по странице товару, у которого
     ещё нет цены;
   - либо просто доставлять вручную те товары, что репортятся в warnings
     (их немного — обычно 1 на категорию).

## Не закрыто / нужно уточнить у заказчика (Бехруз)

- **storeId по 5 точкам** — пока резолвится по имени через `/api/stores/active`
  в `upload_to_crm.py` (fuzzy-match), но алиасы в `config.json` → `store_aliases`
  нужно проверить на реальных названиях точек в CRM.
- **"Мерси Папа", "Бордо", "Испанский чизкейк"** — встречаются в реальных
  WhatsApp-сообщениях по акциям, но физически нет в `Каталог_Pate-2.pdf`.
  Либо это позиции из другого/более старого каталога-документа, либо их нужно
  добавить в `catalog_paste.json` вручную (без фото, пока не найдём картинку).
- **`extract_catalog.py`** не прогнан на полном PDF (только страницы 1-6).
  Нужно:
  1. доделать баг с "осиротевшими" ценами (см. п.4 выше);
  2. прогнать на всех ~26 страницах кондитерских категорий;
  3. визуально проверить результат (особенно картинки — грабли #3);
  4. при необходимости — смержить с уже существующим `catalog_paste.json`
     (там уже руками поправлены все edge-кейсы, не хочется их потерять).
- Категория в CRM "Кондитерские изделия" — id существует, не хардкодим,
  резолвится по имени в `upload_to_crm.py`.
- `upload_to_crm.py` ни разу не тестировался на реальном сервере (у меня
  как у Claude нет доступа к foodsave.kz) — первый реальный прогон обязательно
  через `--dry-run`, затем на 1-2 товарах вручную проверить, что создалось
  в CRM как ожидается, и только потом — массово.
